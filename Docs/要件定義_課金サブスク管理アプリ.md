## 1. アプリ概要

### コンセプト
> 「課金・サブスクを楽しく見える化するアプリ」

家計簿アプリではなく、**趣味消費の見える化**に特化。
ゲーム課金もサブスクも、ポジティブに・楽しく管理できることを目指す。

### アプリ名
- SubLog - サブスク管理アプリ

### ターゲットユーザー

| 優先度 | ターゲット | ニーズ |
|--------|-----------|--------|
| メイン | ソシャゲプレイヤー | ガチャ課金・定期パスを整理したい |
| メイン | サブスク多数契約者 | 何を契約しているか把握したい |
| サブ | オタク・趣味消費者 | 趣味への支出を楽しく記録したい |
| サブ | サブスク断捨離したい人 | 無駄なサブスクを洗い出したい |

### 差別化ポイント
既存の家計簿・サブスク管理アプリとの違いは以下の通り。

- ゲーム課金（ガチャ・週課金・シーズンパス）に**特化した記録機能**
- 「罪悪感」ではなく「楽しみ」として提示する**ポジティブUI・メッセージ**
- ガチャ石・アイテムパックを自分でカスタム登録できる**石プルダウン機能**
- 課金人生の**累積グラフ**（これまでの総課金額）

## 2. 機能要件

### 2-1. コア機能（MVP／v1.0）

#### ① 課金・サブスク登録機能

| 項目 | 内容 | 備考 |
|------|------|------|
| サービス名 | テキスト入力 | 必須 |
| カテゴリ | ゲーム / 音楽 / 動画 / 書籍 / その他 | 選択式 |
| アイコン | 写真ライブラリ or SF Symbols から選択 | |
| 料金 | 金額（円） | |
| 課金タイプ | マンスリー / ウィークリー / シーズンパス / 年額 / 単発 | 選択式 |
| 課金分類 | ガチャ用 / 定期パス / サブスク / その他 | 選択式 |
| 開始日 | 日付ピッカー | |
| 継続状態 | 継続中 / 解約済 / 無料トライアル中 | |
| メモ | 自由テキスト | イベント名など |

#### ② ゲーム課金履歴機能

- 課金記録に「石・アイテム購入」を追加できる
- **ガチャ石カスタム選択肢**：ユーザーが「120石 / 610石 / 10000石」などを事前に自分で登録しておき、記録時はプルダウンから選択するだけでOK
- 一度登録したテンプレートは再利用可能
- 課金記録一覧（日時・金額・内容）の表示

#### ③ ダッシュボード（ホーム画面）

| 表示項目 | 詳細 |
|----------|------|
| 今月の課金合計 | 月ごとに自動集計 |
| 今年の課金合計 | 年ごとに自動集計 |
| 月別課金グラフ | 棒グラフ（Swift Charts） |
| サービス別割合 | 円グラフ（ゲーム / サブスク 等） |
| ポジティブメッセージ | 課金額に応じた換算メッセージ（後述） |
| アクティブなサブスク一覧 | 継続中のサービス一覧サマリー |

#### ④ ポジティブメッセージ機能（「その金額で○○できます」）

**重要な設計方針：課金批判にならないよう、ポジティブなフレーミングとセットにする**

```
例1：今月の課金 ¥6,200
　→「映画館3回分 / カフェ12杯分」
　　 "趣味にこれだけ投資しました！素晴らしい！"

例2：年間課金 ¥84,000
　→「旅行1回分 / ゲーミングデバイス購入も夢じゃない」
　　 "1年間の熱量がここに！"
```

- 換算パターンは複数用意しランダム or スワイプで切り替え
- 「推しへの投資」「楽しんだ証」など課金を肯定するフレーズも用意
- ユーザーが換算パターンを自分で追加できると◎（v2候補）

#### ⑤ サブスク管理機能

| 機能 | 詳細 |
|------|------|
| 一覧表示 | 継続中 / 解約済 タブ切り替え |
| 解約記録 | 解約日・理由メモを記録、履歴として残す |
| 節約額表示 | 解約済みサービスの「解約で節約した金額」を表示 |
| 次回更新日 | 更新日までの残り日数を表示 |

### 2-2. 追加機能（v1.1〜v2.0）

| 機能 | 概要 | 優先度 |
|------|------|--------|
| 更新リマインダー通知 | 更新日の3日前 / 前日にプッシュ通知 | 高 |

## 3. 収益化設計

### フリープラン（無料）

- サービス登録：**上限10件**
- ダッシュボード表示（月別グラフのみ）
- ポジティブメッセージ表示
- スクショ添付：**不可**
- 通知：**不可**

### プレミアムプラン（有料）

| 項目 | 内容 |
|------|------|
| 登録件数 | 無制限 |
| 課金記録 | 無制限 |
| グラフ・統計 | 全種類解放 |
| スクショ添付 | 可能 |
| 更新通知 | 可能 |
| ガチャ石カスタム | 無制限追加 |
| 課金人生グラフ | 解放 |

### 価格帯（目安）

| プラン | 価格 | 備考 |
|--------|------|------|
| 月額 | ¥250〜¥380 / 月 | Appleの価格ティアに合わせて設定 |
| 年額 | ¥1,800〜¥2,400 / 年 | 月額比 約30〜40%オフ |

**実装**：StoreKit 2（Apple公式、iOS 15以上対応）

## 5. データモデル（簡易）

### Service（サービス共通）
```
id: UUID
name: String          // サービス名
category: Enum        // ゲーム / 音楽 / 動画 / 書籍 / その他
icon: Data?           // アイコン画像
billingType: Enum     // マンスリー / ウィークリー / 年額 / 単発
price: Int            // 金額（円）
startDate: Date
isActive: Bool        // 継続中 or 解約済
canceledDate: Date?
memo: String?
```

### PaymentHistory（課金履歴）
```
id: UUID
serviceId: UUID
date: Date
amount: Int
type: Enum            // ガチャ / 定期パス / サブスク更新 / その他
itemName: String?     // 石テンプレート名など
memo: String?
```

### GachaTemplate（石テンプレート）
```
id: UUID
serviceId: UUID
label: String         // "120石", "10000石" など
amount: Int           // 金額
```

### Screenshot（スクショ）※v1.1
```
id: UUID
serviceId: UUID
paymentId: UUID?
imagePath: String
createdAt: Date
```

## 6. 技術スタック

| 分野 | 採用技術 | 理由 |
|------|----------|------|
| 言語 | Swift | iOS標準 |
| UI | SwiftUI | 宣言的UI、モダン |
| データ保存 | SwiftData（iOS 17+）または Core Data | ローカル保存 |
| グラフ | Swift Charts（Apple純正） | iOS 16以上、追加ライブラリ不要 |
| 課金 | StoreKit 2 | Apple公式、iOS 15以上 |
| 画像保存 | FileManager | スクショのローカル保存 |
| 通知 | UserNotifications | 更新リマインダー用 |

> ※ Java経験者がAI（Claude Agent / Codex）に実装を委譲する前提のため、AIに渡す際は「SwiftUI + SwiftData + StoreKit 2 を使って実装して」と明示すると精度が上がります。

## 8. リスク・注意事項

| リスク | 対策 |
|--------|------|
| Apple審査でリジェクト | StoreKit実装は公式ガイドライン厳守。サブスクの説明は明確・日本語でも記載 |
| スクショ保存によるストレージ肥大化 | 画像を圧縮保存（最大1枚あたり200KB程度を目安）、枚数制限も検討 |
| SwiftData vs Core Data の選択 | iOS 17以上のみサポートするなら SwiftData が楽。16以下もサポートするなら Core Data |
| 個人情報・プライバシー | 課金額はセンシティブデータ。ローカル保存のみ＆iCloudバックアップはOS任せで問題なし |
| AI実装の品質管理 | AIへの指示は画面単位で細かく分けて依頼する。まとめて依頼すると品質が下がりやすい |

## 10. 開発の進め方（AI実装前提）

AIエージェント（Claude Agent / Codex）に実装を委譲する際の推奨フロー：

1. **画面単位で指示**する（例：「ダッシュボード画面をSwiftUIで作って」）
2. **データモデルを先に確定**してからUIを依頼する
3. StoreKit 2 は専用の指示ファイルを作って渡す（サンドボックステスト含む）
4. グラフは「Swift Chartsを使って棒グラフを作って」と明示
5. バグが出たらコードごとAIに渡してデバッグを依頼

> 💡 **ヒント**：画面ごとに「画面の目的」「表示するデータ」「ユーザーの操作」を3点セットでAIに伝えると実装精度が大幅に上がります。
